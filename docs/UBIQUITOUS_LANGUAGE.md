# Ubiquitous Language
> Canonical AndThen terms for code, docs, skill prompts, and review reports. Prefer entries that prevent ambiguity, schema drift, or incorrect skill routing.

## Core Terms
| Term | Definition | Avoid |
|------|------------|-------|
| AndThen | The lightweight spec-driven development framework for AI coding agents. Use for the product, plugin, and installed skill bundle as a whole. | workflow toolkit, agent framework |
| Project Document Index | The table in a project's root agent instruction file that maps document types to locations. Skills use it to find and write artifacts. | docs table, path map |
| Project-Specific Guidelines and Rules | Project-local guidance loaded before work to capture conventions, prohibitions, and validation expectations. | local notes |
| Foundational Rules and Guardrails | Universal behavioral rules installed or referenced by AndThen. Non-negotiable across runs. | critical rules, global rules |
| Automation mode | Non-interactive behavior enabled by `--auto`: conservative assumptions, no prompts, stop only on real contract failures. | headless orchestration |

## Requirements and Planning
| Term | Definition | Avoid |
|------|------------|-------|
| Requirements Clarification | Discovery artifact from the `andthen:clarify` skill, usually `requirements-clarification.md`. Resolves scope, decisions, edge cases, and open questions. | clarification doc |
| Feature Implementation Specification (FIS) | Execution-sized specification generated by the `andthen:spec` skill or `andthen:plan` skill. Contains a Feature Overview and Goal (Intent + Expected Outcomes), Required Context when needed, Acceptance Scenarios, Structural Criteria, Scope & Boundaries, an Implementation Plan, and optional Final Validation Checklist items. | spec, feature spec, implementation spec |
| Intent (FIS) | One-sentence statement under `## Feature Overview and Goal` naming why the feature exists – the problem solved or user/business value unlocked. Distinct from feature title and scope summary. | feature description, summary |
| Expected Outcome | FIS-internal, behavioral, user-/business-observable success condition under `## Feature Overview and Goal`, tagged `[OC<NN>]`. Distinct from upstream PRD outcomes (inlined into Required Context) and from Structural Criteria (non-behavioral invariants). | PRD outcome, success criterion, structural criterion |
| Outcome tag | `[OC<NN>]` token on an Acceptance Scenario that anchors the scenario to the Expected Outcome(s) it exemplifies. Two-digit zero-padded, comma-joined when a scenario tags multiple outcomes. | scenario tag, OC reference |
| Required Context | Optional FIS section containing load-bearing upstream spans inlined verbatim and source-pinned at spec time. Authoritative for execution. | background, references |
| Deeper Context | Optional FIS section containing supplementary `path#anchor` pointers to read only when Required Context leaves a gap. | required reading, extra context |
| Plan Bundle | Co-located planning directory containing `prd.md`, `plan.json`, generated FIS files, and optional supporting assets. | implementation plan, plan folder |
| plan.json | Local typed runtime plan written by the `andthen:plan` skill and read by execution, ops, review, and routing skills. The `--from-issue` flow materializes one at `.agent_temp/from-issue-<N>/plan.json`. Not a "ledger" – that term is reserved for the Reconciliation Ledger. | plan.md, markdown plan, plan table, ledger |
| GitHub Issue Transport | Markdown rendering of a local artifact into a GitHub issue body for handoff or visibility. The local JSON or markdown artifact remains canonical unless the mode says otherwise. | GitHub source of truth |
| Story | Vertical, bounded, verifiable unit in `plan.json` that maps 1:1 to a FIS. | task, ticket, work item |
| Story Catalog | The `stories[]` collection in `plan.json`, or its markdown table rendering in GitHub Issue Transport. | story list, plan table |
| Phase | Logical story group in the plan overview, identified by IDs such as `P1`. | milestone, stage |
| Wave | Execution ordering group within a phase, identified by IDs such as `W1`; stories in the same wave may run together when dependencies allow. | batch, sprint |
| sourceRefs | `plan.json` story field for PRD feature IDs, anchors, and upstream requirement references used to seed FIS context. Markdown label: `Source refs`; prose: source refs. | Source Refs as a JSON field, citations |
| assetRefs | `plan.json` story field for wireframes, ADRs, design-system references, or other upstream assets the FIS author needs. Markdown label: `Asset refs`; prose: asset refs. | Asset Refs as a JSON field |
| bindingConstraints | `plan.json` top-level array of verbatim PRD spans that must flow unchanged into FIS Required Context. Markdown/prose section: `Binding Constraints`; singular entry: binding constraint. | hard requirements |
| sharedDecisions | `plan.json` top-level array of inter-story interface or design contracts referenced by producing and consuming stories. Markdown/prose section: `Shared Decisions`; singular entry: shared decision. | cross-story notes |
| riskSummary | `plan.json` top-level array tying stories to risk levels and mitigations. Markdown/prose section: `Risk Summary`. | risk table, risk log |
| FIS Provenance | The FIS header fields `**Plan**:` and `**Story-ID**:` linking a plan-backed FIS to its source plan and story. | provenance header |

## Execution and Review
| Term | Definition | Avoid |
|------|------------|-------|
| Stop-the-Line | Execution discipline that treats objective red gates as work to finish before advancing or claiming completion. | fail-fast, stop work |
| Objective Red Gate | Binary failing check such as build, test, lint, type-check, wiring check, or task Verify line. | blocker when the failure is an objective check |
| Subjective Finding | Review or visual-validation finding that requires judgment and gets one focused remediation pass before escalation. | review issue |
| Traceability Gate | Requirement that tests and motivated code changes trace to FIS requirements or explicitly appended discovered requirements. | trace check |
| Discovered Requirement | Requirement found during execution and appended through the sanctioned FIS channel before dependent tests or code are written. | inferred requirement |
| Finding | Concrete review issue with severity, evidence, and remediation guidance. | comment, issue when not referring to GitHub |
| Findings Filter | Calibration pass that challenges candidate findings before final severity and report inclusion. | triage pass |
| Gap Review | Review lens comparing implementation against a requirements baseline and emitting the PASS/FAIL contract. | requirement review |
| Council Review | Multi-perspective review using selected specialist reviewers, debate, Findings Filter, and synthesis. | multi-reviewer review |
| Remediate Findings | Workflow run by the `andthen:remediate-findings` skill to re-validate findings, apply minimal fixes, re-verify, and update workflow state. | address comments |
| Tech Debt Backlog | Durable backlog for deferred validated findings that cannot be fixed in the current remediation pass. | parking lot |
| Reconciliation Ledger | Per-FIS, append-only document at `{fis-without-ext}.reconciliation-ledger.md` recording OPEN/CLOSED/WITHDRAWN reconciliation entries when an amendment leaves an upstream doc stale. Mutated only via the `andthen:ops` skill's `update-ledger`. The only persisted document called a "ledger" (the Run Ledger is in-memory). | drift log, plan.json (which is a plan, not a ledger) |
| Run Ledger | In-memory aggregate (`completed`/`failed`/`skipped`/`blocked_by`) the `andthen:exec-plan` skill keeps during a multi-story run to feed the aggregate report. Distinct from the persisted `plan.json` and from the Reconciliation Ledger. | run log |

## Protocols
| Term | Definition | Avoid |
|------|------------|-------|
| Interactive-by-Contract | Rule that the `andthen:clarify` skill must include user-answered discovery questions before producing its artifact. | optional interactivity, headless clarify |
| Recommend, don't decide | Clarification posture: offer a defensible recommended answer while requiring the user to ratify or redirect it. | assume the recommendation |
| Named Output Blocks | Shared execution protocol for `CONFUSION:`, `MISSING REQUIREMENT:`, and `NOTICED BUT NOT TOUCHING:`. | ad hoc status notes |
| ASSUMPTION: | Automation-mode record of the safest defensible interpretation chosen when interactive clarification is unavailable. | silent default |
| Proof-of-Work | FIS authoring rule that each Acceptance Scenario's Given/When/Then contract has a concrete proof path, and each Structural Criterion is proved by a task Verify line. | proof note |
| Prove-It Pattern | Test-first bugfix flow where a failing test proves the defect before the fix makes it pass. | regression test after fix |
| Anti-Cheat Invariant | TDD rule that a proof test cannot be deleted, disabled, or weakened to make the build green. | weaken the test |
| Status-Write Contract | Multi-story execution rule centralizing plan, FIS, and state writes to prevent double-writing shared status artifacts. | bookkeeping rule |

## Distribution
| Term | Definition | Avoid |
|------|------------|-------|
| Skill Bundle | Self-contained installed skill directory containing its prompt plus copied references, templates, scripts, and metadata. | skill package |
| Shared Plugin Asset | Canonical file under `plugin/references/` consumed by multiple skills and inlined during generic/user-tier installs. | shared reference |
| Canonical Reference | Source-of-truth copy of shared content before install-time inlining or path rewrites. | shared doc |
| Install-time Propagation | Installer behavior that copies skills, inlines shared assets, rewrites namespace references, and adjusts path tokens for the target tier. | install rewrite |
| Documentation Lookup Agent | Plugin-tier `documentation-lookup` agent, with prefixed generated/user-tier variants, used for external documentation retrieval. | docs agent |
| Review Persona Agent | Review-scoped agent under `plugin/agents/review-*.md`, used by council mode and Critic review for focused reviewer perspectives. | generic review agent, broad agent zoo |
| Worktree Isolation | Optional execution or review isolation using separate git worktrees for parallel or full-fidelity work. | worktree mode |

## Overloaded Terms
| Term | Context A | Meaning A | Context B | Meaning B |
|------|-----------|-----------|-----------|-----------|
| spec | General discussion | Any specification-like document | AndThen planning | Prefer FIS for the execution artifact generated by the `andthen:spec` skill |
| plan | Skill/prose | The `andthen:plan` skill or planning act | Artifact | `plan.json`, the typed local runtime plan |
| story | Product requirements | User story in a PRD or clarification artifact | Plan bundle | A `stories[]` entry that maps 1:1 to a FIS |
| review | Skill | The unified workflow run by the `andthen:review` skill | Report content | Findings, verdict, and evidence produced by a lens or chain |
| agent | User environment | An AI coding agent such as Claude Code, Codex, Aider, or Cursor | AndThen plugin tier | The `documentation-lookup` agent plus review persona agents under `plugin/agents/review-*.md` |
| context | Prompt/runtime | Model context or fresh-session context | FIS | Required Context and Deeper Context sections carrying upstream intent |
| context | Domain design | Bounded Context in DDD | Project discovery | Project context loaded from root agent instructions |
| state | Workflow artifact | Project State document for session continuity | Plan status | `stories[].status` values inside `plan.json` |
| issue | GitHub integration | A GitHub issue used as input or transport | Review | A finding; prefer Finding to avoid confusion |
| asset | Install system | Shared Plugin Asset under `plugin/references/` | Plan schema | `assetRefs` pointing to upstream assets |
| ledger | Reconciliation | The per-FIS Reconciliation Ledger (the preferred, canonical meaning) | exec-plan run | The in-memory Run Ledger; never `plan.json`, which is a plan |

## Usage Notes
| Term | Preferred usage |
|------|-----------------|
| `andthen:<name>` references | Write "the `andthen:<name>` skill" or "the `andthen:<name>` agent"; do not describe a skill invocation as spawning a sub-agent. |
| Feature Implementation Specification (FIS) | Define once, then use FIS. |
| plan.json | Use for the local typed plan; reserve `plan.md` for legacy migration or GitHub Issue Transport discussion. Never call it a "ledger". |
| Plan schema fields | Use exact camelCase for JSON fields (`sourceRefs`, `assetRefs`, `bindingConstraints`, `sharedDecisions`, `riskSummary`). Use markdown labels only in GitHub Issue Transport. |
